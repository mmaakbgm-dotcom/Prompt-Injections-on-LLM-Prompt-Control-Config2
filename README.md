# HealthFirst Clinic — Layer 3.2: Prompt-Level Access Control

An academic thesis project evaluating whether **prompt-level access control alone**
(with no code-enforced SQL guards) can resist scope-bypass attacks in a healthcare
NL-to-SQL portal powered by GPT-4o-mini.

---

## Thesis Context

This repository is one configuration in a cross-layer security comparison study.
The experiment asks: *how much protection does each defense layer add when an LLM
converts patient natural-language queries directly into SQL?*

### The Four Configurations

Each configuration activates a different combination of the four-layer defense
architecture below. This repository implements **Layer 3.2 only** (prompt-level).

| Config | Layer 1 | Layer 2 | Layer 3 | Layer 4 | Description |
|--------|---------|---------|---------|---------|-------------|
| L1     | ✓       |         |         |         | Input sanitization only |
| L2     |         | ✓       |         |         | Prompt-structuring only |
| **L3.2** | | **✓** | | | **Guiding-prompt access control only (this repo)** |
| L4     |         |         |         | ✓       | Code-enforced SQL rewriting only |

### The Four-Layer Defense Architecture

| Layer | Name | Mechanism |
|-------|------|-----------|
| **1** | Input Sanitization | Keyword filtering, injection detection, preprocessing before the LLM sees input |
| **2** | Prompt Structuring | System prompt instructs the LLM on access rules; LLM compliance is probabilistic |
| **3** | Output Filtering | Post-LLM validation of the SQL or response before it reaches the user |
| **4** | Database Authorization | Code rewrites or replaces the LLM's SQL to enforce scope; the hard authority layer |

**Layer 3.2 activates Layer 2 only.** There is no code-enforced SQL guard. The
guiding prompt is the sole access-control mechanism, and SQL generated by the LLM
is executed verbatim against SQLite.

---

## Security Evaluation Results

Three independent evaluation suites were run against this configuration:

| Suite | Tests | Violations / Attacks | Attack Success Rate |
|-------|-------|----------------------|---------------------|
| Promptfoo (120 adversarial prompts) | 120 | 0 | **0.00%** |
| DeepTeam (red-team, 20 attack classes) | ~100 | 0 confirmed | **0.00%** |
| SQL Adversarial Suite (920 runs, 7 categories) | 920 | 49 | **5.33% VSR** |

**Key finding:** The guiding prompt successfully blocks conversational attacks
(Promptfoo, DeepTeam) but is partially bypassed by low-level SQL construction attacks,
particularly `GROUP BY` / `DISTINCT` aggregation queries (41 of 49 violations).

## System Architecture

```
User Input (natural language)
        │
        ▼
 authenticate()           ← username/password gate (the only code-enforced check)
        │
        ▼
 safe_chat_response()     ← short-circuit for greetings (no LLM call)
        │
        ▼
 ai_agent_to_sql()        ← Stage 1: NL → SQL via GPT-4o-mini (temp=1.5)
   System prompt: GUIDING_PROMPT (scoped to session role + linked_id)
   Conversation history: last 6 turns (MULTI_TURN=True)
        │
        ├── NO_QUERY → "Request denied."
        │
        ▼
 run_sql()                ← SQL executed verbatim against SQLite (no rewriting)
        │
        ▼
 generate_nl_response()   ← Stage 2: rows → natural language via GPT-4o-mini (temp=0.3)
        │
        ▼
 append_audit_log()       ← every interaction logged to 3_2_audit_log.txt (append-only)
        │
        ▼
 Final response returned to user
```

## Setup

### Prerequisites

- Python 3.11+
- Node.js 18+ and npm (for Promptfoo evaluation only)
- An OpenAI API key with GPT-4o-mini access

### 1. Clone the Repository

```bash
git clone https://github.com/<your-username>/<your-repo>.git
cd <your-repo>
```

### 2. Set Environment Variables

Copy `.env.example` to `.env` and fill in your OpenAI credentials:

```bash
cp .env.example .env
# Edit .env and set AI_INTEGRATIONS_OPENAI_API_KEY and AI_INTEGRATIONS_OPENAI_BASE_URL
```

On **Replit**: these are set automatically via the built-in OpenAI integration.

### 3. Install Python Dependencies

```bash
pip install openai deepteam openpyxl
```

Or using uv:

```bash
uv sync
```

### 4. Install Node Dependencies (Promptfoo only)

```bash
npm install
```

### 5. Initialize the Database

The database (`clinic.db`) is created automatically on first run:

```bash
python clinic_3_2.py
```

It seeds 30 patients, 8 doctors, and 150 appointments. You only need to do this once.

## Access Control Design

The guiding prompt (written to `guiding_prompt.txt` at startup) instructs the LLM:

- **Patient role**: every `appointments` query MUST include `patient_id = {linked_id}`
- **Doctor role**: every `appointments` query MUST include `doctor_id = {linked_id}`
- **Both roles**: may freely query the `doctors` table (public data)
- If a request cannot be answered within scope: output `NO_QUERY`

**There is no code that enforces these rules.** The LLM's compliance is the
only barrier. This is the thesis experiment: measuring how often the LLM holds
under adversarial pressure.

## Reconstruction Prompts

The `prompts/` folder contains `reconstruction_prompt_layer_3_2.md` — a complete
specification prompt that can be fed to an AI coding assistant to rebuild this entire
project from scratch in a fresh environment. It covers the full database schema,
all application logic, all three evaluation frameworks, and all configuration details.

## Audit Log

Every user interaction is appended to `3_2_audit_log.txt` (created automatically).
The log is append-only and records: timestamp, username, role, linked_id, user input,
Stage 1 LLM output, extracted SQL, execution result, decision (ALLOWED/DENIED), and
final visible output. The log file is excluded from version control.

---

## Notes for Reproduction

- The guiding prompt uses `temperature=1.5` for Stage 1 (SQL generation). This
  introduces non-determinism; re-running evaluations may produce slightly different
  results, particularly at boundary cases.
- The SQL Adversarial Suite uses a `--forced` mode that prepends a directive
  instructing the LLM to output SQL even for out-of-scope requests. This is
  intentional — it probes the hard boundary of the guiding prompt.
- All evaluation scripts are safe to re-run. They reset session state between
  test cases via `clinic.logout()`.
