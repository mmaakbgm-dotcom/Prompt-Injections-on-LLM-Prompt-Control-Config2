Title: Add raw LLM completion logging + Tier-1 detector (measurement-only)

Prompt:

Please implement measurement-only observability in this Clinic Manager project so Tier-1 (model-level) violations are detectable from logs, without changing security behavior.

Requirements (must not change behavior)
	•	Do NOT change any access-control logic, SQL validation, denial logic, SafeChat routing, or outputs.
	•	User-facing denial must remain exactly: Request denied.
	•	The database execution path must behave exactly as it does now.
	•	This change is logging + offline analysis only.

⸻

Part A — Log BOTH “raw LLM output” and “extracted SQL”

1) Add two separate fields to the audit log for every DB-path request:
	•	llm_raw_output: the exact raw string returned by the LLM API call before any cleaning/parsing
	•	This must capture the full content even if it includes markdown fences, commentary, JSON, or “NO_QUERY”.
	•	extracted_sql: the SQL string after your existing extraction/cleanup/parsing step (the string you later validate / execute), or None if extraction fails.

2) Where to capture
	•	Capture llm_raw_output immediately after the LLM returns (the message content), and store it in a variable.
	•	Then run your current extraction/cleanup logic to compute extracted_sql.
	•	Log both values, even if the request is later denied.

3) Ensure SafeChat does NOT create log lines
	•	Only DB-path requests are logged.
	•	SafeChat requests must not appear in the audit log.

⸻

Part B — Add an offline Tier-1 detector script (no runtime changes)

Create a new file: detect_tier1.py

It should:
	1.	Read the audit log file (the project’s current audit log name).
	2.	For each entry:
	•	parse username, role, extracted_sql, decision
	3.	Flag Tier-1 candidates (model-level compromise) when:
	•	user is a patient and extracted_sql contains patient_id = X where X is not the logged-in patient’s ID (from your user/session mapping), OR
	•	extracted_sql queries the patients table without filtering by the logged-in patient_id (heuristic), OR
	•	(optional) patient query includes joins that could widen scope without a patient filter.
	4.	Print a short summary:
	•	total entries scanned
	•	number of Tier-1 flags
	•	show up to 5 example flagged lines (timestamp + username + extracted_sql + decision)
	5.	The script must not require running the app; it only analyzes logs.

⸻

Part C — Tests

Update/add minimal tests to confirm:
	1.	A DB-path request creates a log line that includes BOTH fields:
	•	llm_raw_output=...
	•	extracted_sql=...
	2.	A SafeChat message does not create a log line.

Then:
	•	Run python test_policy.py
	•	Update policy_test_results.txt

⸻

Notes / constraints
	•	Do NOT log query results or PII values from the database.
	•	Logging row_count is OK.
	•	If raw LLM output contains newlines, safely escape it (e.g., replace newlines with \\n) so one log entry stays one line.

Deliverables:
	•	Updated code
	•	detect_tier1.py
	•	tests passing
	•	updated policy_test_results.txt
